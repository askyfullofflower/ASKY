# 基于数字孪生的档案馆温湿度实时监测与智能调控系统
## 本科毕业答辩介绍（详细版 · PPT 适用）

## 一、研究背景与问题痛点

### 1.1 文化遗产保护的环境挑战

纸质档案是不可再生的文化遗产，对保存环境极为敏感。**国家标准 DA/T 15—2023** 对档案库房环境有严格规定：

| 指标 | 国标要求 | 目标值 |
|------|---------|-------|
| 温度 | 14 ~ 24°C | **20°C** |
| 相对湿度 | 45 ~ 60%RH | **50%RH** |
| PM2.5 | ≤ 35 μg/m³ | — |
| NOx | ≤ 0.10 mg/m³ | — |

### 1.2 传统人工巡检的三大痛点

| 痛点 | 具体问题 |
|------|---------|
| **响应滞后** | 人工巡检周期长，异常发现通常滞后数小时 |
| **覆盖盲区** | 多楼层、多库房难以实现全天候无死角监控 |
| **决策依赖经验** | 缺乏量化数据支撑，调控操作依赖个人经验 |

### 1.3 数字孪生的解决方案

> 数字孪生技术将物理空间映射为**实时可交互的虚拟副本**，通过感知→建模→分析→反馈的闭环，实现毫秒级自动响应和全方位智能管控。

本系统在此基础上进一步接入 **大语言模型（LLM）** 与**飞书群聊机器人**，将监控感知、智能决策与即时通知打通为一体。

│   传感器仿真 → PID-Fuzzy调控 → 14步主循环广播                    │
│   ├── AI代理接口 → LongCat-Flash-Chat 大模型                    │

### 2.2 监控覆盖范围

- **7 个独立库房**，分布于 F1~F4 楼层（库房 A-G）

- 每秒广播一次完整状态快照，所有客户端**毫秒级同步**

## 三、后端核心设计

### 3.1 传感器仿真引擎

后端维护 **31 条基准样本**构成的正常数据池，采样时加入平滑限速（温度 ±0.4°C/tick、湿度 ±1.0%RH/tick），避免数据跳变。

系统启动时**回溯填充 720 个历史点**（约 12 分钟），利用正弦叠加随机扰动公式生成仿真历史曲线，保证前端图表一上线即有连续历史可看：

$$T[i] = 21.5 + 2.0 \cdot \sin(i \times 0.02) + \varepsilon_T$$

$$H[i] = 52.0 + 5.0 \cdot \sin(i \times 0.015 + 1) + \varepsilon_H$$

### 3.2 PID-Fuzzy 智能调控算法

系统核心算法：引入**模糊增益切换**的 PID 控制思想，实现"偏差越大调控越猛、偏差越小越精细"的自适应响应。

**温度调控逻辑：**

$$e_T = T_{当前} - 20°C, \quad K_p = \begin{cases} 25 & |e_T| > 3°C \text{（强力响应）} \\ 15 & \text{否则（精细调节）} \end{cases}$$

$$P_{AC} = \min\!\left(100,\ |e_T| \times K_p\right)$$

**湿度调控逻辑：** 除湿机 $K_p = 10$（偏差 > 10%RH）或 5；加湿器对称处理。

**通风系统协同联动：** 当任一主设备功率 > 50% 时，通风自动拉至 80% 辅助气流循环。

算法输出后**离散化为四档**（off / low 30% / medium 65% / high 100%），防止设备频繁微调损耗。

### 3.3 设备物理效应模型（闭环关键）

每个 tick 叠加设备实际物理效应，形成真实的**传感器-算法-设备**闭环：

| 设备 | 最大温度效果 | 最大湿度效果 |
|------|------------|------------|
| 空调（制冷/制热） | ±0.8°C/tick × (power/100) | 微弱 ±0.1%RH/tick |
| 除湿机 | — | −1.5%RH/tick × (power/100) |
| 加湿器 | — | +1.5%RH/tick × (power/100) |
| 通风 | ±0.05°C/tick | ±0.1%RH/tick |

> **闭环数据流：** 传感器采样 → PID计算 → 设备输出 → 物理效应叠加 → 下次采样

### 3.4 9 种异常注入场景

支持一键触发，用于系统测试与演示：

| 场景 | 效果描述 | 告警触发 |
|------|---------|---------|
| 🔥 火灾模拟 | 随机一库房 +1.2°C/tick，PM2.5 同步飙升 | 立即严重 |
| 🌨 寒潮 | 全馆 −1.0°C/tick | 温度低限告警 |
| 🌧 梅雨 | 全馆 +1.8%RH/tick | 湿度高限告警 |
| 🏜 干燥 | 全馆 −1.8%RH/tick | 湿度低限告警 |
| 💧 漏水 | 局部风险值 = clamp(45+tick×8, 35, 100)，≥60% 触发 detected | 漏水告警 |
| ☁ 有害气体 | PM2.5 +12+tick×1.1，NOx 同步升高 | 空气质量告警 |
| 🌡 持续高温 | 全馆 +0.5°C/tick | 温度高限告警 |
| 📊 越界采样 | 按比例注入 20% 越界数据 | 数据异常告警 |
| ✅ 清除异常 | 立即恢复正常波动 | — |

### 3.5 三级告警与自动升级机制

**告警生命周期：**

```
[新建] active ──(处置中)──→ processing ──(审核)──→ review ──(完成)──→ resolved
         │
         └── 超过 180s 未处置 ──→ 等级自动升1级 ──→ 触发飞书推送
```

| 等级 | 颜色标识 | 升级条件 |
|------|---------|---------|
| 一般（L1） | 黄色 | active 超 180s → 升至较高 |
| 较高（L2） | 橙色 | active 超 180s → 升至严重 |
| 严重（L3） | 红色 | 同时触发飞书机器人推送 |

- **120 秒去重**：相同类型告警不重复推送，避免消息轰炸
- **排名推送**：推送内容包含偏差最大的 **4 个库房排名**

### 3.6 每 1 秒广播的 14 步主循环

```
① 采样传感器数据（含平滑限速）
② 叠加异常效果（inject_* 逻辑）
③ 执行 PID-Fuzzy 调控计算
④ 施加设备物理效应
⑤ 计算漏水检测状态
⑥ 计算空气质量（PM2.5 / NOx）
⑦ 计算各库房独立温湿度
⑧ 楼层聚合统计（floor_stats）
⑨ 同步阈值告警记录
⑩ 检查升级逻辑
⑪ 飞书通知判定与发送
⑫ 追加历史记录（MySQL / deque）
⑬ 线性趋势预测（下一小时温湿度预报）
⑭ WebSocket 广播到所有连接客户端
```

## 四、飞书群聊机器人实时推送（重点）

### 4.1 推送架构设计

后端通过 **HTTP Webhook** 直连飞书群聊机器人，无需额外中间件，异常触发后**秒级到达**管理员群聊：

```
后端 14步主循环
   └─ ⑪ 飞书通知判定
         ├── 条件1：告警等级升至"严重"
         ├── 条件2：发现新的高危异常（火灾/漏水/有害气体）
         └── 条件3：环境指标持续越界 120s 以上
                  │
                  ▼  HTTP POST（异步非阻塞）
         飞书 Flow Webhook → 飞书群聊机器人 → 管理员手机即时弹通知
```

### 4.2 推送消息格式

飞书机器人消息采用**富文本结构化格式**，关键信息一目了然：

```
🚨 档案馆环境告警通知

告警等级：🔴 严重
告警类型：温度超标
告警时间：2026-05-15 14:32:07

📊 库房偏差排名（TOP 4）：
  1. 档案库房E（F3）  超标 +4.8°C  当前 24.8°C
  2. 档案库房D（F2）  超标 +3.2°C  当前 23.2°C
  3. 档案库房G（F4）  超标 +2.1°C  当前 22.1°C
  4. 档案库房A（F1）  超标 +1.4°C  当前 21.4°C

🏭 当前设备状态：空调制冷 High (100%)，通风 High (80%)
⏱ 本次告警已持续 187 秒，等级已自动升级

请及时登录系统处置 → http://[内网地址]:3000
```

### 4.3 推送触发机制详解

| 触发条件 | 具体逻辑 | 推送频率控制 |
|---------|---------|-----------|
| **首次告警创建** | 新建严重级（L3）告警时立即推送 | — |
| **告警自动升级** | 告警在 active 状态超 180s，等级升1级后推送 | 每次升级推送1次 |
| **高危异常注入** | 注入火灾/漏水/有害气体时即时推送 | 触发即推送 |
| **指标持续越界** | 同一指标越界超 120s 且未推送过 | 120s 去重窗口 |

### 4.4 去重与防轰炸设计

为避免在极端异常时频繁推送骚扰管理员，系统实现了多层去重机制：

- **时间窗口去重**：同一告警键（库房 + 类型）120s 内不重复推送
- **等级去重**：仅在等级变化时触发推送，稳定在某一等级不反复发送
- **批量合并**：同一时刻触发多个告警时，合并为一条消息，附带 TOP4 排名

## 六、PC 端三维可视化

### 6.1 Three.js 3D 场景

PC 端为单一 HTML 文件（约 **6400 行**），内含完整 CSS 设计系统 + Three.js 3D 场景 + Chart.js 图表，零依赖部署。

建筑体块用 `ExtrudeGeometry` 从 2D 平面挤出，材质采用 `MeshPhysicalMaterial`，配置 `transmission`（半透明）、`clearcoat`（清漆）、`sheen`（丝绒边缘光），呈现科技感玻璃建筑效果。

**7 层光照系统：**

| 光源 | 颜色 | 说明 |
|------|------|------|
| HemisphereLight | 天蓝/深海蓝 | 半球散射基底 |
| AmbientLight | 蓝紫 | 全局填充 |
| DirectionalLight × 2 | 冷白/天蓝 | 主光投影 + 背光补偿 |
| PointLight coolFill | 青色 | 科技感冷蓝点光 |
| 扫描点光 A/B | 青/橙 | sin/cos 轨道运动，动态扫场 |
| selectedRoomGlowLight | 暖白 | 选中库房跟随点光 |

**库房状态脉冲动画**（每帧执行）：

$$I = base + \frac{\sin(t \cdot speed + \phi)+1}{2} \cdot amp$$

正常模式慢呼吸（speed=1.2），温度告警快速闪烁（speed=4.2），**一眼可识别异常区域**。

**开场动画**：2200ms 缓动，相机从 `(0, 228, 430)` 飞入 `(0, 168, 258)`，easeOutCubic 曲线。

### 6.2 温湿度场云图（IDW 插值）

设计了**书架探测点模拟**：每个探测点温度偏差由 **8 个物理因子叠加**：

```
tempDelta =
  状态偏置（temp_alert → +1.3°C）
  + 门口气流影响
  + 角落滞留效应（+0.55）
  + 书架遮挡散热减弱（-0.42）
  + 空调出风口高斯羽流（±0.9）
  + 空间位置梯度（nx×0.34 - ny×0.24）
  + 书架间通道气流波（±0.24）
  + 大尺度对流波动 + 随机噪声（±0.62）
```

用 **IDW 反距离加权插值**（power=1.9）渲染连续热力场：

$$val = \frac{\sum w_i v_i}{\sum w_i}, \quad w_i = \frac{1}{d_i^{1.9}}$$

温度映射 HSL 蓝→红，湿度映射 HSL 青→深蓝，可切换显示。

### 6.3 报警追溯与安全防护

- 所有服务端数据插入 DOM 前强制经过 `escHtml()` 做 HTML 实体转义，防止 **XSS 注入**
- 报警面板**签名防重渲染**机制：字段未变化时跳过 DOM 重建，避免高频闪烁
- PC 端与小程序同步告警状态，处置操作双端实时回显

## 七、微信小程序

### 7.1 防崩溃容错架构

`app.js` 用 try/catch 包裹 `realtime-store.js` 的加载，失败时自动创建 `fallbackStore`（所有方法为空操作），确保**应用先启动、再暴露真实错误**，杜绝"白屏启动失败"。

### 7.2 WebSocket 多地址自动重试

实现 **5 路候选地址自动重试**（上次成功地址 → 手动配置 → 局域网 IP → 127.0.0.1 → localhost），连接超时 5 秒自动切换下一候选，成功后写入本地存储记忆。HTTP API 同样具备**七级降级策略**，极大提升不同网络环境下的连通成功率。

### 7.3 五大功能页面

| 页面 | 核心功能 | 技术亮点 |
